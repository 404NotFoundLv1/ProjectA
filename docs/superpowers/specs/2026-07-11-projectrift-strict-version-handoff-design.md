# ProjectRift 严格小版本交付制设计

## 1. 目的

本文定义 Codex 接管 ProjectRift 后续开发时的固定协作协议。目标是在尽量减少用户手工操作的同时，确保每个小版本独立实现、独立验证、由用户亲自验收和提交，并且未经用户明确授权绝不开始后续版本。

本文只定义开发协作、项目隔离、Git 边界、验证门禁和交付方式，不改变游戏设计文档及版本路线图中的玩法范围。

## 2. 当前基线

- UE 工程路径：`E:\MyWork\ProjectA\ProjectA.uproject`
- 产品代号：ProjectRift
- C++ 模块名：`ProjectA`
- 引擎版本：Unreal Engine 5.8
- 当前分支：`main`
- 当前已完成版本：v0.4.7
- 下一个候选版本：v0.4.8 Vertical Slice 稳定化、回归保护与打包基线
- 产品需求来源：完整游戏设计文档 v1 和完整生产至 Steam 路线图 v2

仓库中的真实代码、类名、蓝图引用和数据资产始终优先于文档中的示例命名。不得为了匹配文档示例而批量重命名现有模块、类、资产或接口。

## 3. 不可违反的操作边界

### 3.1 单项目隔离

Codex 只能操作 `E:\MyWork\ProjectA` 及其明确属于 ProjectA 的本地服务、构建进程和输出目录。

Codex 不得读取、检查、停止、修改、借用或控制任何其他项目；不得把其他项目的 UE 编辑器、MCP 服务、日志、进程或资产当作 ProjectA 的工具或上下文。

ProjectA 的 Unreal MCP 固定使用：

```text
http://127.0.0.1:8001/mcp
```

端口 `8000` 不属于 ProjectA。Codex 不得向端口 `8000` 发送 ProjectA 请求，也不得为了释放端口而干预其他进程。若 ProjectA 的 `8001` 服务不可用，Codex 必须停止 UE MCP 操作并报告，不得自动尝试其他项目或其他已监听端口。

### 3.2 严格小版本门禁

只有用户明确发出“开始 vX.Y.Z”或等价指令后，Codex 才能开始对应小版本。

Codex 每次只能实现一个小版本。当前小版本完成后必须立即停止，不得自动规划、预实现、重构或开始下一个版本。后续版本只有在用户验收当前版本、完成自己的 Git 提交，并再次明确发出开始指令后才能启动。

### 3.3 Git 边界

所有开发暂时直接在现有 `main` 分支进行。

Codex 不得：

- 创建或切换 Git 分支；
- 执行 `git add` 或其他暂存操作；
- 创建、修改或重写提交；
- 创建 Git 标签；
- 推送远端；
- 创建 Pull Request；
- stash、reset、checkout、restore 或丢弃用户修改；
- 重写或 squash 历史。

Codex 可以使用只读 Git 命令检查当前分支、工作区状态、差异和历史。每个版本开始前必须检查工作区；若发现与当前任务重叠的用户未提交修改，Codex 应停止并说明冲突，不得覆盖或绕过。

版本完成后的验收和提交由用户亲自执行。Codex只提供建议提交信息。

## 4. 版本执行状态机

每个小版本严格遵循以下状态：

```text
等待用户开始指令
-> 只读预检
-> 当前版本设计与实施计划
-> 实现当前版本
-> 编译和自动化验证
-> UE/联机/打包验收准备
-> 交付报告
-> 强制停止
-> 用户亲自验收
-> 用户亲自提交
-> 等待下一条开始指令
```

任何失败均留在当前版本内处理，不得通过开始下一版本来规避当前问题。

## 5. 单版本工作流

### 5.1 开始条件

开始一个小版本必须同时满足：

- 用户明确指定版本号或版本名称；
- 当前分支为 `main`；
- 已读取该版本在路线图中的目标、编码范围、禁止事项和验收标准；
- 已完成只读仓库审计；
- ProjectA MCP 若为本版本所需，则必须确认连接的是 `127.0.0.1:8001`；
- 未发现会被本版本覆盖的用户修改。

若开始条件不满足，Codex应报告具体阻塞项，不得猜测、跨项目寻找替代服务或扩大修改范围。

### 5.2 只读预检

Codex 在修改前必须确认：

- 真实模块名、类名前缀和目录结构；
- 现有 GameMode、GameState、PlayerState、GameInstance 和 Subsystem 职责；
- GAS、背包、任务、刷怪、撤离、结算和旅行相关接口；
- 当前蓝图、Widget、地图和 DataAsset 对 C++ 类型的引用；
- 相关自动化测试、构建脚本和日志；
- 当前版本与已完成 v0.4.7 的边界；
- 拟修改文件和潜在存档、复制、旅行、Cook 风险。

### 5.3 实现原则

- 只实现当前小版本明确要求的行为。
- 不顺便实现后续版本，不做无关重构。
- 优先兼容扩展现有接口；不破坏旧蓝图、资产和存档引用。
- 网络状态由服务端权威决定，客户端只发送请求和播放表现。
- 数据驱动优先使用稳定 ID、GameplayTag、DataAsset、DeveloperSettings 和明确结构体。
- 新增持久化字段必须配套版本提升、迁移和失败路径。
- 新增网络事务必须考虑重复 RPC、断线、重连和幂等。
- 新日志使用项目自己的 LogCategory，并在 Shipping 中关闭开发作弊入口。
- 不使用 UE 5.8 Experimental 功能作为游戏运行时核心依赖；Unreal MCP 仅作为编辑器开发工具。

### 5.4 UE MCP 操作规则

Codex 可以通过 ProjectA 的官方 Unreal MCP 对当前版本所需的蓝图、DataAsset、地图、材质、对象属性和自动化测试执行受控操作。

每次二进制资产操作必须：

- 在操作前读取目标资产及其引用上下文；
- 只修改当前版本需要的资产；
- 操作后编译或校验资产；
- 明确保存已验证的资产；
- 在交付报告中列出所有改变的 `.uasset` 和 `.umap`；
- 不批量覆盖、重命名或删除资产；
- 不通过脚本伪造 Unreal 二进制资产文件。

### 5.5 编译与测试

验证强度必须与修改风险相匹配：

- 普通 `.cpp` 函数体修改：执行目标编译和相关自动化测试；
- `.h`、UCLASS、USTRUCT、UENUM、GAS 属性、FastArray、复制字段、Build.cs、Target.cs 或插件修改：保存 ProjectA 资产，关闭 ProjectA 编辑器后执行完整编译，再重新打开 ProjectA；
- 单机功能：运行相关自动化测试和单人 PIE 验收；
- 网络功能：增加 2 Players Listen Server 验收；
- 关键里程碑：按路线图要求执行 4 人、打包或实机验证；
- 打包相关功能：验证 Cook、默认地图、启动、完整闭环和日志。

无法自动化的主观画面、手感、多人体验或硬件兼容项目必须转化为清晰、可重复的用户验收步骤。

## 6. 错误与阻塞处理

遇到编译失败、测试失败、MCP 错误或行为异常时，Codex必须留在当前版本中进行根因调查。

Codex 不得：

- 用删除测试或降低断言来制造通过结果；
- 在没有复现证据时重构整个系统；
- 为规避当前问题而切换到下一版本；
- 访问其他项目寻找可复用的运行中 MCP 服务；
- 未经用户允许执行不可逆外部操作。

若需要用户参与，Codex只请求完成当前阻塞所必需的最小操作，并说明原因、风险和完成后如何验证。

## 7. 版本完成标准

只有同时满足以下条件，Codex 才能声明当前小版本“待用户验收”：

- 当前版本路线图中的编码范围已完成；
- 没有实现被明确排除的后续内容；
- 相关编译成功；
- 相关自动化测试通过；
- 可自动执行的 UE 验收已完成；
- 无已知 Blocker 或 Critical 问题；
- 其他已知问题具有复现步骤、影响范围和规避方式；
- 存档、复制、旅行、旧蓝图和 Cook 风险已说明；
- 所有修改文件和二进制资产已列出；
- 工作区保持未暂存、未提交状态；
- 未开始下一小版本。

## 8. 固定交付报告

每个小版本完成后，Codex 必须交付一份面向用户的报告，包含：

1. 版本号与完成结论；
2. 修改文件和资产清单；
3. 核心实现说明；
4. 编译命令与结果；
5. 自动化测试命令、通过数与失败数；
6. 已由 Codex 完成的 UE MCP 验证；
7. 用户需要亲自执行的验收步骤；
8. 已知问题与风险；
9. 建议 Git commit 信息；
10. 明确声明“已停止，未开始下一版本”。

Codex 不得在交付报告后继续修改项目，除非用户要求修复当前版本的验收问题或明确开始另一个版本。

## 9. 需要用户保留的最终决策

以下事项不能由 Codex 静默替代用户决定：

- 每个小版本的最终验收；
- Git 提交、标签、推送和发布；
- 玩法手感和主观美术质量；
- 第三方资产购买及许可证接受；
- Steamworks 凭据、真实 AppID、商店提交、价格、法务和发布日期；
- 对外发布、上传、删除云端数据或其他不可逆外部操作。

## 10. ProjectA MCP 隔离落地方案

在开始 v0.4.8 之前，先完成一次仅限 ProjectA 的开发环境配置：

- 将 ProjectA 的 Unreal MCP 自动启动端口设为 `8001`；
- 将 ProjectA 的 Codex MCP URL 设为 `http://127.0.0.1:8001/mcp`；
- 在仓库根目录添加长期 `AGENTS.md`，写入本文的单项目隔离、main 分支、用户提交和严格版本门禁规则；
- 重启 ProjectA 编辑器和 Codex 后，验证 `list_toolsets` 来自 ProjectA；
- 不探测、停止或更改占用其他端口的项目和进程。

该配置完成并通过连接验证后，ProjectA 才进入 v0.4.8 的只读预检。

## 11. 首个交付里程碑

首个候选交付为 v0.4.8。它只负责冻结和稳定现有 v0.4.7 Vertical Slice，包括幂等结算、生命周期清理、关键 Smoke Test、诊断日志、Development 打包配置和单人/2 人闭环验证。

v0.4.8 不新增职业、敌人、装备、制造、章节或正式 UI，也不重新设计现有任务、背包、结算、刷怪或旅行架构。完成后 Codex 必须按照本文规定停止，等待用户验收和提交。
