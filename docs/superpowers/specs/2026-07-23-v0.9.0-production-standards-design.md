# ProjectRift v0.9.0 Production Standards Design

## Status

Approved approach: Scheme A — add a ProjectA-only editor module and keep all production validation out of packaged runtime builds.

## Goal

Freeze ProjectRift's art, content-directory, naming, performance, licensing, and chapter-level interface contracts before production asset volume increases. New production assets must be traceable to a declared source, and ProjectRift maps must be machine-checkable without modifying Epic template or Variant content.

## Scope and boundaries

- Operate only inside `E:\MyWork\ProjectA`.
- Add one `ProjectAEditor` C++ module loaded only by the ProjectA editor target.
- Do not create another Unreal project or another editor application.
- Do not rename or reorganize existing template, Variant, Blueprint, map, or reflected runtime assets.
- Hard-fail production validation only for `/Game/ProjectRift` and explicitly declared ProjectRift production maps.
- Treat `/Game/Characters`, `/Game/ThirdPerson`, `/Game/Variant_*`, engine content, and plugin content as external dependencies. Validate dependency policy but do not modify them.
- Keep the profile schema at v9 and make no gameplay or save migration changes.
- Do not claim that an unprovided license is valid. Missing or unknown licensing data fails closed.
- Do not generate production models, textures, music, fonts, or other AI-authored art in this version.

## Architecture

### Runtime/editor separation

`ProjectA` remains the only runtime game module. A new `ProjectAEditor` module depends on `ProjectA`, `UnrealEd`, `AssetRegistry`, `AssetTools`, `DataValidation`, and the editor-only mesh, texture, audio, and map inspection APIs. `ProjectAEditor.Target.cs` loads both modules, while the game target and Shipping packages load only `ProjectA`.

The editor module exposes:

- a reusable production-validation service;
- a commandlet-style unattended entry point used by local automation;
- automation tests that exercise the same service;
- structured JSON and human-readable Markdown reports under `Saved/Automation/ProjectRiftProductionValidation`.

No validator type is referenced by runtime classes, cooked assets, save data, or gameplay code.

### Authoritative policy files

Policy is stored as reviewable UTF-8 files:

- `Config/ProjectRift/ProductionValidation.json` — scan roots, allowed external roots, class-to-prefix rules, per-asset budgets, dependency rules, level contracts, warning/failure severities.
- `Config/ProjectRift/AssetLicenseRegistry.json` — authoritative licensing declarations.
- `docs/projectrift/production/` — human-facing art, naming, import, level, performance, audio, and licensing guides.

The validator rejects an unknown schema version and malformed or duplicate records. Configuration cannot silently relax a required field.

## License registry

Each exact production asset package, or an explicitly allowed folder-owned declaration, records:

- `AssetPath`
- `SourceKind`: `ProjectOwned`, `EpicProvided`, `Marketplace`, `Contractor`, `Purchased`, or `Other`
- `SourceUri`
- `AuthorOrVendor`
- `CommercialUse`: `Allowed`, `Denied`, or `Unknown`
- `Modification`: `Allowed`, `Denied`, or `Unknown`
- `Attribution`: `Required`, `NotRequired`, or `Unknown`
- `AttributionText`
- `LicenseDocumentPath`
- `InvoicePath`
- `ReviewedBy`
- `ReviewedDate`
- `Notes`

Only `ProjectOwned` and `EpicProvided` may omit invoice data. All source kinds require an evidence path or an explicit Epic/project-owned declaration. `CommercialUse=Unknown`, `Modification=Unknown`, or `Attribution=Unknown` is a blocking result for new production art. The tool validates completeness and file/path consistency; it does not provide legal advice.

Existing ProjectRift code/data-only assets can be covered by a Project-owned folder declaration. Imported meshes, textures, sounds, music, fonts, animations, and materials require exact asset records.

## Content and naming policy

New ProjectRift content must live under an approved `/Game/ProjectRift/<Domain>` directory. Nothing new may be added directly to `/Game`, `/Game/ProjectRift`, or unrelated template/Variant roots.

The initial prefix contract includes:

- maps `L_`
- Blueprints `BP_`
- data assets `DA_`
- static meshes `SM_`
- skeletal meshes `SK_`
- textures `T_`
- materials `M_`
- material instances `MI_`
- Niagara systems `NS_`
- Niagara emitters `NE_`
- sound waves `S_`
- sound cues `SC_`
- sound mixes `SMix_`
- sound classes `SClass_`
- widgets `WBP_`
- animations `A_`, montages `AM_`, animation Blueprints `ABP_`
- curves `Curve_`

Names must be ASCII, contain no spaces, remain within the configured length limit, and avoid generated suffixes such as `_New`, `_Copy`, or `_Duplicate`.

## Asset validation

Validation produces stable rule IDs and asset paths. Blocking rules cover:

- illegal directory or missing/incorrect prefix;
- missing required license entry;
- texture dimensions above their class budget or non-power-of-two dimensions where the policy requires it;
- production static meshes without usable simple collision;
- static meshes without Nanite or an acceptable LOD chain according to configured triangle thresholds;
- direct use of a base material where the domain contract requires a material instance;
- audio assets without a ProjectRift sound class and required concurrency/routing metadata;
- forbidden hard references from production content into Developer, test, or Variant content;
- hard references across domain boundaries that are not allowlisted;
- production maps missing required level-interface entries.

Unused assets and suspicious reverse-reference gaps are warnings only in v0.9.0. The validator never deletes or moves assets.

## Performance budget

The policy defines conservative graybox/first-production budgets rather than platform-final tuning:

- texture maximum dimensions by semantic group;
- static-mesh triangle bands and required Nanite/LOD behavior;
- material-slot and texture-sampler guidance;
- skeletal-mesh bone/material guidance;
- audio duration/compression/category guidance;
- per-map actor, light, shadow-caster, decal, Niagara, and always-loaded hard-reference budgets.

Budgets are documented with rationale and are configurable without recompiling the validator. Exceeding a hard ceiling blocks validation; approaching a ceiling emits a warning.

## Chapter-level interface

Production chapter maps declare a contract in `ProductionValidation.json` with stable IDs and expected actor classes/tags for:

- `SpawnGroups`
- `ObjectiveSockets`
- `ExtractionSocket`
- `BossArena`
- `StreamingPartitions`

The validator reuses authoritative ProjectRift actors where they already exist:

- spawn groups use `APREnemySpawnPoint` and non-empty `SpawnGroupId`;
- active objective anchors use `APRRiftObjectiveActor` subclasses with stable objective node IDs;
- extraction uses `APRExtractionZone`;
- Boss arenas use `APRBossEncounterController`;
- streaming checks use the map/world-partition data actually authored in Unreal.

No parallel runtime socket hierarchy is introduced unless a production map proves the existing interface insufficient. A dedicated editor-only validation fixture map under `/Game/ProjectRift/Developer/Maps` exercises every contract without changing accepted gameplay maps. `L_Rift_Guardian` is also registered as the first production-contract audit target, but the validator reports its findings before any map correction is considered.

## User workflow

Developers can run validation from:

- ProjectA editor validation/menu integration;
- ProjectRift local PowerShell automation;
- an unattended editor command suitable for CI.

Every run writes:

- overall pass/fail;
- schema and project version;
- rule ID, severity, asset/map path, message, and remediation hint;
- inventory counts and license coverage;
- performance and reference summaries.

The source registry and report never copy license documents into cooked game content.

## Error handling and safety

- Invalid policy or registry JSON blocks the run with the exact file and parse error.
- Missing asset packages, missing evidence paths, and duplicate license declarations are deterministic failures.
- Assets outside ProjectA are never opened through another editor or MCP service.
- Validation is read-only. It does not save, rename, move, delete, fix, or resubmit assets.
- The editor module must be absent from Shipping binaries and cooked module descriptors.

## Documentation deliverables

Create:

- asset naming and directory standard;
- import checklist and source-file retention policy;
- collision, LOD, Nanite, material-instance, texture, and audio rules;
- performance budget table;
- chapter-level interface guide;
- asset license registry guide and evidence checklist;
- a one-page visual style/color/lighting brief template.

The visual brief is a production template, not AI-generated art. The user supplies or approves final visual references and license documents during manual acceptance.

## Testing and acceptance

Test-first automation covers:

- editor module isolation and Shipping exclusion;
- valid and malformed policy/registry files;
- all required license fields, unknown permissions, duplicate coverage, and missing evidence;
- directory and class-prefix rules;
- texture, mesh, material, audio, dependency, and unused-asset classifications;
- level-contract success and each missing interface category;
- stable JSON/Markdown reporting;
- `L_ProductionValidation_Test` passing all blocking rules;
- `L_Rift_Guardian` producing no unapproved blocking findings;
- project version `0.9.0`, profile schema v9, and existing ProjectRift regressions.

Final verification includes focused editor tests, the full ProjectRift suite, Development package and LocalSmoke, Shipping BuildCookRun, Shipping `-NullRHI` startup, editor-module exclusion, and the shared `UnrealEditor.modules` preservation check.

Manual acceptance verifies the visual brief, registered licenses/evidence supplied by the user, editor validation UX, report readability, and a deliberately invalid fixture that demonstrates the validator catches missing level points and license data.

## Out of scope

- Producing final art, audio, fonts, animations, or lighting.
- Legal approval of third-party licenses.
- Bulk-renaming, moving, deleting, or auto-fixing existing assets.
- Retrofitting all Epic template or Variant content.
- Runtime asset validation, downloadable-content rules, source-control hooks, or Steam depot configuration.
- v0.9.1 lobby, tutorial, or visual repair work.
